# 登录超时配置说明

## 概述

本文档详细说明了前端和后端的登录超时时间配置，以及两者的同步机制。

## 后端配置（OAuth2 Server）

### 配置文件位置
`XunLuAdminCloud/oauth2-server/src/main/java/org/wm/config/ClientInitializer.java`

### 各客户端的超时配置

#### 1. 内部管理后台客户端 (xunlu-admin-web)

**客户端ID**: `xunlu-admin-web`
**使用场景**: 本系统的 React 前端登录
**授权模式**: 密码模式 (Password Grant)

```java
.tokenSettings(TokenSettings.builder()
    .accessTokenTimeToLive(Duration.ofHours(2))    // Access Token: 2小时
    .refreshTokenTimeToLive(Duration.ofDays(7))    // Refresh Token: 7天
    .reuseRefreshTokens(false)                     // 不重用refresh token
    .accessTokenFormat(OAuth2TokenFormat.SELF_CONTAINED)  // 使用JWT
    .build())
```

**超时时间总结**:
- **Access Token 有效期**: 2 小时（7200 秒）
- **Refresh Token 有效期**: 7 天（604800 秒）
- **Token 重用策略**: 每次刷新生成新的 refresh_token

#### 2. 演示第三方应用客户端 (demo-third-party-app)

**客户端ID**: `demo-third-party-app`
**使用场景**: 演示外部应用如何接入本认证平台
**授权模式**: 授权码模式 (Authorization Code + PKCE)

```java
.tokenSettings(TokenSettings.builder()
    .accessTokenTimeToLive(Duration.ofMinutes(30))  // Access Token: 30分钟
    .refreshTokenTimeToLive(Duration.ofDays(30))    // Refresh Token: 30天
    .reuseRefreshTokens(false)
    .accessTokenFormat(OAuth2TokenFormat.SELF_CONTAINED)
    .build())
```

**超时时间总结**:
- **Access Token 有效期**: 30 分钟（1800 秒）
- **Refresh Token 有效期**: 30 天（2592000 秒）

#### 3. 微服务客户端 (microservice-internal)

**客户端ID**: `microservice-internal`
**使用场景**: 服务间调用认证
**授权模式**: 客户端凭证模式 (Client Credentials)

```java
.tokenSettings(TokenSettings.builder()
    .accessTokenTimeToLive(Duration.ofHours(1))  // Access Token: 1小时
    .accessTokenFormat(OAuth2TokenFormat.SELF_CONTAINED)
    .build())
```

**超时时间总结**:
- **Access Token 有效期**: 1 小时（3600 秒）
- **Refresh Token**: 不适用（客户端凭证模式不使用 refresh token）

## 前端配置

### 配置文件位置
`frontend-cloud/src/utils/auth.ts`

### Token 过期检测逻辑

```typescript
export function isTokenExpired(): boolean {
  const expireTime = getTokenExpireTime();
  if (!expireTime) return true;
  // 提前5分钟刷新token
  return Date.now() > expireTime - 5 * 60 * 1000;
}
```

**关键配置**:
- **提前刷新时间**: 5 分钟（300 秒）
- **刷新策略**: 在 token 过期前 5 分钟自动刷新

### Token 过期时间的保存

前端在登录成功后，会保存后端返回的 `expires_in` 值：

```typescript
// 保存token过期时间
if (msg?.expires_in) {
  setTokenExpireTime(msg.expires_in);
}
```

`setTokenExpireTime` 函数会将过期时间转换为绝对时间戳：

```typescript
export function setTokenExpireTime(expiresIn: number) {
  // 保存token过期时间戳（当前时间 + 过期秒数）
  const expireTime = Date.now() + expiresIn * 1000;
  window.localStorage.setItem(TOKEN_EXPIRE_TIME_KEY, expireTime.toString());
}
```

## 前后端超时时间一致性分析

### ✅ 完全一致

**前后端的超时时间配置是一致的**，具体体现在：

1. **后端返回准确的过期时间**
   - 后端在 OAuth2 Token 响应中返回 `expires_in` 字段（单位：秒）
   - 对于 `xunlu-admin-web` 客户端，返回值为 `7200`（2 小时）

2. **前端使用后端返回的值**
   - 前端不硬编码超时时间
   - 登录成功后，直接使用后端返回的 `expires_in` 值
   - 保存为绝对时间戳：`当前时间 + expires_in * 1000`

3. **同步机制保证**
   - 后端修改 token 有效期后，前端会自动使用新的值
   - 无需修改前端代码

### 实际超时时间表（针对前端登录场景）

| 时间点 | 说明 | 剩余有效时间 |
|-------|------|------------|
| 登录成功 | 获得 access_token | 2 小时 |
| 1小时55分钟后 | 前端自动刷新 token | 提前 5 分钟 |
| 刷新成功 | 获得新的 access_token | 重新计时 2 小时 |
| 如果未刷新 | 2小时后 access_token 过期 | 0 |
| 遇到 401 | 使用 refresh_token 刷新 | 最多 7 天内有效 |

## 自动刷新机制

### 1. 预防性刷新（请求前拦截器）

在 `app.tsx` 的 `authHeaderInterceptor` 中：

```typescript
// 检查token是否过期，如果过期且开启了自动登录，则尝试刷新token
if (token && isTokenExpired() && getAutoLogin()) {
  if (!isRefreshing) {
    isRefreshing = true;
    const refreshSuccess = await tryRefreshToken();
    isRefreshing = false;
    // ...
  }
}
```

**触发时机**: Access Token 剩余有效时间 < 5 分钟

### 2. 补救性刷新（响应拦截器）

在 `app.tsx` 的 `demoResponseInterceptors` 中：

```typescript
if (data.code === 401) {
  // 尝试使用refresh_token刷新access_token
  const autoLogin = getAutoLogin();
  const refreshTokenValue = getRefreshToken();

  if (autoLogin && refreshTokenValue && !isRefreshing) {
    isRefreshing = true;
    const refreshSuccess = await tryRefreshToken();
    isRefreshing = false;
    // ...
  }
}
```

**触发时机**: 收到 401 错误响应

### 3. 应用启动自动登录

在 `app.tsx` 的 `getInitialState` 中：

```typescript
if (!token && refreshTokenValue && autoLogin) {
  console.log('尝试使用refresh_token自动登录');
  const refreshSuccess = await tryRefreshToken();
  // ...
}
```

**触发时机**: 应用启动时没有 access_token 但有 refresh_token

## 用户体验时间线

### 正常使用场景

```
用户登录
  ↓
[0-1h55m] 正常使用，token 有效
  ↓
[1h55m] 前端检测到即将过期，自动刷新
  ↓
[1h55m-3h55m] 继续正常使用（新 token）
  ↓
[3h55m] 再次自动刷新
  ↓
持续使用最多 7 天...
```

### 离开后返回场景

```
用户登录并勾选"自动登录"
  ↓
使用 30 分钟后关闭浏览器
  ↓
第二天打开浏览器
  ↓
检测到没有 access_token
  ↓
检查有 refresh_token（还在 7 天内）
  ↓
自动使用 refresh_token 获取新的 access_token
  ↓
无需重新输入密码，直接进入系统
```

### Token 完全过期场景

```
用户登录并勾选"自动登录"
  ↓
7 天后访问系统
  ↓
refresh_token 已过期
  ↓
自动刷新失败
  ↓
清除所有认证信息
  ↓
跳转到登录页，需要重新登录
```

## 配置建议

### 当前配置评估

✅ **优点**:
1. Access Token 短期有效（2小时），降低泄露风险
2. Refresh Token 长期有效（7天），提供良好的用户体验
3. 提前 5 分钟刷新，用户无感知
4. 前后端完全同步，无配置不一致风险

⚠️ **可优化项**:
1. 可以根据业务需求调整 Refresh Token 有效期（7天可能较短）
2. 可以考虑根据用户角色设置不同的超时时间
3. 可以添加"记住我"时长配置（如 30 天）

### 推荐配置方案

#### 方案 1: 高安全场景（金融、医疗等）
```java
.accessTokenTimeToLive(Duration.ofMinutes(30))   // 30分钟
.refreshTokenTimeToLive(Duration.ofDays(1))      // 1天
```

#### 方案 2: 平衡场景（当前配置）
```java
.accessTokenTimeToLive(Duration.ofHours(2))      // 2小时
.refreshTokenTimeToLive(Duration.ofDays(7))      // 7天
```

#### 方案 3: 便捷场景（内部系统）
```java
.accessTokenTimeToLive(Duration.ofHours(4))      // 4小时
.refreshTokenTimeToLive(Duration.ofDays(30))     // 30天
```

## 修改配置的方法

### 修改后端超时时间

编辑 `ClientInitializer.java` 中的相应配置：

```java
.tokenSettings(TokenSettings.builder()
    .accessTokenTimeToLive(Duration.ofHours(4))     // 修改为4小时
    .refreshTokenTimeToLive(Duration.ofDays(30))    // 修改为30天
    .reuseRefreshTokens(false)
    .accessTokenFormat(OAuth2TokenFormat.SELF_CONTAINED)
    .build())
```

**注意**: 修改后需要重启 OAuth2 服务，已存在的客户端配置会自动更新。

### 修改前端刷新提前时间

编辑 `frontend-cloud/src/utils/auth.ts`：

```typescript
export function isTokenExpired(): boolean {
  const expireTime = getTokenExpireTime();
  if (!expireTime) return true;
  // 修改提前刷新时间（如改为10分钟）
  return Date.now() > expireTime - 10 * 60 * 1000;
}
```

**建议**: 提前刷新时间应该在 5-10 分钟之间，太短可能导致频繁 401，太长浪费 token 有效期。

## 常见问题

### Q1: 为什么有时候会提示"登录已过期"？

**A**: 可能的原因：
1. Refresh Token 已过期（超过 7 天）
2. 未勾选"自动登录"
3. 手动清除了浏览器缓存
4. 后端重启导致 Redis 中的 token 失效（如果使用 Redis）

### Q2: 修改后端超时时间后，前端会自动同步吗？

**A**: 会的。前端使用后端返回的 `expires_in` 值，无需修改前端代码。

### Q3: 可以为不同用户设置不同的超时时间吗？

**A**: 当前配置是按客户端（应用）级别的。如果需要按用户级别配置，需要：
1. 在用户表中添加超时配置字段
2. 自定义 Token 生成器，根据用户信息设置过期时间
3. 前端相应调整

### Q4: Access Token 和 Refresh Token 可以设置为相同的过期时间吗？

**A**: 技术上可以，但不推荐。推荐配置：
- Access Token: 短期（分钟到小时级别）
- Refresh Token: 长期（天到周级别）
- Refresh Token 有效期 >= Access Token 有效期 * 2
